### 1. 概述

之前我们已经了解了 Java 运行时的内存大致布局，其中程序计数器、虚拟机栈和本地方法栈都是**线程私有**的，生命周期与线程相同，同生同灭。同时这些内存区域的分配情况与类结构高度相关，基本在编译器就可以确定下来，因此这些区域中的内存回收不是我们要优先考虑的重点。

而 Java 堆和方法区则拥有很高的不确定性，这不仅仅是因为这些区域是**全局共享**的——一个接口的多个实现类所占用的内存是不同的，方法内的不同分支导致的内存开销也是不确定性，绝大多数时候只有在运行期我们才能确认程序究竟会创建使用多少对象。这部分的内存是**动态**的，不是所有的对象都跟程序有这相同的生命周期，如果不加以控制仍由其混乱增长，那么内存溢出就是迟早的事。

**垃圾收集器**所关注的正是这部分内存该如何管理。

说起**垃圾收集（Garbage Collection，下文简称 GC**），有不少人把这项技术当作 Java 语言的伴生产物。事实上，垃圾收集的历史远远比 Java 久远，在 1960 年诞生于麻省理工学院的 Lisp 是第一门开始使用内存动态分配和垃圾收集技术的语言。

但无论我们处于何种时代，垃圾回收器都需要思考以下三件事情：

- ** [[#3. 如何回收？| 如何回收？]] **

### 2. 哪些内存需要回收？
#### 2.1 堆内存垃圾收集
在堆里面存放着 Java 世界中几乎所有的对象实例，垃圾收集器在对堆进行回收前，第一件事情就是要确定这些对象之中哪些还“存活”（也就是被 [[Java 引用关系#^ecffbd | 引用]]）着，哪些已经“死去”（“死去”即不可能再被任何途径使用的对象）了。

##### 2.1.1 引用计数法

首先最朴素的想法就是对在对象中添加一个引用计数器：每当有一处地方（引用）用到了这个对象，就自增；引用失效了（比如函数退出）就自减。任何时刻下计数器为零的对象就是不可能再被使用了。

只需要一点点的额外空间，就可以很轻松地实现最基本的垃圾处理与判定了。但是只有计数器是远远不够的，如果对象之间拥有互循环引用，那么计数器将永不为 0，哪怕互引用的对象不再会被别的对象引用。现代主流的 JVM 虚拟机也不再采用这种方式作为垃圾分类的算法。再额外引入一些算法或许能解决这个问题，但不如整体换一个思路。

##### 2.1.2 可达性分析
目前主流语言的内存管理都是通过**可达性分析（Reachability Analysis）**来判定对象是否存活。基本思路是只从系统中的根对象开始，通过引用关系向下搜索，经过的路径称为“**引用链（Reference Chain）**”。如果某个对象是不可达的，那么此对象就再无被使用可能。

>  通过可达性分析判定出不可达的对象之后，并不意味着立刻“判决死刑”：
>  
> 首先经过可达性分析之后，所有不可达对象被标记。同时进行一次筛选，条件是此对象是否有必要执行 `finalize()` [^1]方法。如果对象需要执行 `finalize()` 方法，那么该对象将会被放置在一个名为 **F-Queue** 的队列之中“缓刑”，而不是被立即回收之后由一条虚拟机自动建立的、低调度优先级的 **Finalizer** 线程去执行[^2]它们的 `finalize()` 方法。然后 JVM 会记录对象的 `finalize()` 方法执行情况，此后无论多少次 GC 检测到该对象不可达，都会跳过 `finalize()`  直接回收。
> 
> 对象**只有一次**逃离死亡队列的机会，那就是在 ` finalize() ` 方法中对象重新与引用链上的任意对象建立关联（比如将 ` this ` 指针赋值给其他类变量），这时对象在 `finalize()` 执行后会被移出垃圾回收的集合。
> 
> 总结：**一个对象的 `finalize()` 最多只被系统自动调用一次，逃离 GC 的自救机会也仅有一次**。

#### 2.2 方法区垃圾收集
通常情况下方法区（又可以叫永久代）中是不强求实现垃圾回收机制的（因为性价比很低），但是大多数垃圾收集器也是支持对这快内存空间进行垃圾收集。方法区中垃圾收集的目标主要为已不再使用的常量和类。

常量回收和对象回收类似，比如有一个字符串常量“Java”存在于常量池中，但系统内没有任何字符串对象的值是“Java”，也就是不再被引用，那么这个常量就会被移出常量池。

而判定一个不再使用的类就更复杂了，通常需要满足三个条件：
- 该类所有的实例都已经被回收，如果有子类也应该都被回收。
- 加载该类的类加载器已经被回收。
- 该类对应的 `java.lang.Class` 对象没有在任何地方被引用，不能有对此对象的反射使用。

三个条件都满足后，类对象便会被标记为可回收。具体关于是否进行回收，则可以通过 `-Xnoclassgc` 参数控制（HotSpot）。

[^1]: 在较新版本的 Java （JDK 9+）中 `finalize()` **已被标记为废弃**（`@Deprecated`）。应该使用 `try-with-resources` 或 `Cleaner` 类而不是 `finalize()`
