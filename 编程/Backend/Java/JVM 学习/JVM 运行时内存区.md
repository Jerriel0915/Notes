## 2. 运行时内存区域
### 2.1 程序计数器
在 JVM 中**程序计数器（Program Counter Register）**是一块较小的内存空间，与 CPU 芯片中的 PC 寄存器类似，它可以看作是当前线程所执行的字节码的行号指示器。因此，每个线程内部都有一个独立的程序计数器，各线程之间互不干扰，是线程**私有**的内存区域。

不过，JVM 中的计数器只对虚拟机内部有效，如果当前执行的是本地（Native）方法，这个计数器的值应该为空或是没有意义（Undefined）。

### 2.2 Java 虚拟机栈

^516a4d

与程序计数器一样，**Java 虚拟机栈（Java Virtual Machine Stack）** 也是线程**私有**的，它的生命周期 与线程相同。

这块内存区域描述的是 Java 方法执行时的线程内存模型：每个方法被执行的时候，Java 虚拟机都会同步创建一个**栈帧（Stack Frame）** 用于存储局部变量表、操作数栈、动态连接、方法出口等信息。每一个方法被调用直至执行完毕的过程，就对应着一个栈帧在虚拟机栈中从入栈到出栈的过程。

局部变量表中存放编译期可知的各种 JVM 基本数据类型、对象引用类型和 `returnAddress` 类型（指向了一条字节码指令的地址），这些数据类型在表中的存储空间以**局部变量槽（Slot）**来表示。局部变量槽的物理内存大小由具体虚拟机自行决定。

规范规定，如果线程请求的栈深度大于虚拟机所允许的深度，比如递归深度过高，将抛出 `StackOverflowError` 异常，也就是常说的“爆栈了”；如果 Java 虚拟机栈容量可以动态扩展[^1]，当栈扩展时无法申请到足够的内存会抛出 OOM—— `OutOfMemoryError` 异常，通常是申请的线程过多导致的。

### 2.3 本地方法栈
**本地方法栈（Native Method Stacks）**与虚拟机栈所发挥的作用差不多，正如其名，其只为虚拟机使用到的本地（Native） 方法服务。

### 2.4 Java 堆
通常在 Java 程序中，堆空间占用的内存是最大的。**堆（Java Heap）** 空间被所有线程**共享**，这是因为此区域内存放的是**对象实例**，而几乎所有的 Java 对象实例都在这里分配内存。

> "The heap is the runtime data area from which memory for all class instances and arrays is allocated."——《Java 虚拟机规范》

堆空间也是垃圾收集器管理的内存区域，所以也可以叫它 **GC 堆（Garbage Collected Heap）**。从垃圾收集器的额角度来看待这片区域，又通常可以细分为“新生代”“老年代”“永久代”等区域，但无论是哪个区域，存储的都只能是对象的实例。将 Java 堆 细分的目的只是为了更好地回收内存，或者更快地分配内存。

规范规定，Java 堆 可以处于物理上不连续的内存空间中，但在逻辑上它应该被视为连续的。这很好理解，操作系统在存储文件时就是这么干的。

如果在 Java 堆 中没有足够内存完成实例分配，并且堆也无法再扩展时，JVM 将会抛出 OOM 异常。

### 2.5 方法区
与[[#2.4 Java 堆|堆]]类似，**方法区（Method Area）** 也是被各个线程共享的区域，用于存储已被虚拟机加载的类型信息、常量、静态变量、即时编译器编译后的代码缓存等数据。

在 HotSpot 虚拟机上，方法区是使用**永久代**来实现的，而其余的一些虚拟机实现则没有永久代的概念。好处是可以让虚拟机像管理 Java 堆那样管理这块内存而不用编写新的逻辑，但坏处则是让 Java 应用更容易出现内存溢出[^2]。

规范规定，方法区也可以和 Java 堆一样不需要连续的内存区域，甚至可以不进行垃圾回收。数据进入永久代并不代表着“永久”存在了，只是相对来说永久代中的数据极少会触发垃圾回收。此区域内的垃圾回收主要针对常量池的回收和对类型的卸载。

当方法区无法满足新的内存分配需求时，将抛出 `OutOfMemoryError` 异常。

### 2.6 运行时常量池
**运行时常量池（Runtime Constant Pool）** 是方法区的一部分。
