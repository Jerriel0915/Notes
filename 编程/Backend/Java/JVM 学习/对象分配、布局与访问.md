本文以 HotSpot 虚拟机为具体例子，粗解在 [[JVM 运行时内存区#2.4 Java 堆|Java 堆]] 中对象分配、布局和访问的全过程。

### 1. 对象的创建
当 Java 虚拟机遇到一条字节码 new 指令时，首先将去检查这个指令的参数是否能在常量池中定位到 一个类的符号引用，并且检查这个符号引用代表的类是否已被加载、解析和初始化过。如果没有，那 必须先执行相应的类加载过程。

类加载检查通过后便需要为新对象分配内存。类加载完成后，对象所需的内存空间便可完全确定，因此分配内存的工作就是在 Java 堆中找到一块合适大小的内存块，并将其划分出去。

不妨先假定 Java 堆在物理上是连续的内存空间，那么根据内存管理的不同可以分成两个情景：

- 如果堆中区域是**规整**的（比如使用像 Serial、ParNew 等带 Compact 过程的收集器时），被严格分为使用区和空闲区，中间以一个哨兵指针作为分界点，那么分配内存再简单不过了，只需要将指针往空闲区的方向移动一定长度就行了。这种方式被称为**“指针碰撞”（Bump The Pointer）**。
- 如果堆中区域并**不规整**（比如使用像 CMS 这类基于 Sweep 算法的收集器），被使用的内存和空闲内存混杂在一块，那么就不能简单的使用一个指针来区分了。这个问题同样出现在操作系统的文件管理系统上，如果通过遍历整块硬盘来找到合适的空间碎片存放文件的话那就太慢了！二者的解决方案可以说是相同的——额外维护一个空间分配的列表，记录哪些空间区域是可用的，哪些是已经被占用的，然后再找[^1]一块合适的空间划分出来，同时更新列表上的记录。这种方式被称为**“空闲列表”（Free List）**。[^2]

> [!NOTE] 取长补短
> 虽然说采用类似于 CMS 收集器时采用的是较复杂的空闲列表法，但为了更快地分配，CMS 设计了一个叫做 `Linear Allocation Buffer` 的分配缓冲区：缓冲区之间由空闲列表管理，而缓冲区内部则使用指针碰撞方式分配内存。
> 而对于更现代的 JVM 来说，则会使用更为复杂的 **Region-based** 内存管理架构，思想上也是二种方式的取长补短。




[^1]: 至于怎么找到这块“合适”的空间，可以采用很多方法如首次适应、最佳适应和最坏适应等，这并不是我们讨论的重点。

[^2]: 当然，操作系统中不止这一种方法，还有空闲链表、位图、成组链接等方法
