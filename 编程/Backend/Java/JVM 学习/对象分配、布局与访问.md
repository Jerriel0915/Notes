本文以 HotSpot 虚拟机为具体例子，粗解在 [[JVM 运行时内存区#2.4 Java 堆|Java 堆]] 中对象分配、布局和访问的全过程。

### 1. 对象的创建
当 Java 虚拟机遇到一条字节码 new 指令时，首先将去检查这个指令的参数是否能在常量池中定位到 一个类的符号引用，并且检查这个符号引用代表的类是否已被加载、解析和初始化过。如果没有，那 必须先执行相应的类加载过程。

#### 1.1 分配内存
类加载检查通过后便需要为新对象分配内存。类加载完成后，对象所需的内存空间便可完全确定，因此分配内存的工作就是在 Java 堆中找到一块合适大小的内存块，并将其划分出去。

不妨先假定 Java 堆在物理上是连续的内存空间，那么根据内存管理的不同可以分成两个情景：

- 如果堆中区域是**规整**的（比如使用像 Serial、ParNew 等带 Compact 过程的收集器时），被严格分为使用区和空闲区，中间以一个哨兵指针作为分界点，那么分配内存再简单不过了，只需要将指针往空闲区的方向移动一定长度就行了。这种方式被称为**“指针碰撞”（Bump The Pointer）**。
- 如果堆中区域并**不规整**（比如使用像 CMS 这类基于 Sweep 算法的收集器），被使用的内存和空闲内存混杂在一块，那么就不能简单的使用一个指针来区分了。这个问题同样出现在操作系统的文件管理系统上，如果通过遍历整块硬盘来找到合适的空间碎片存放文件的话那就太慢了！二者的解决方案可以说是相同的——额外维护一个空间分配的列表，记录哪些空间区域是可用的，哪些是已经被占用的，然后再找[^1]一块合适的空间划分出来，同时更新列表上的记录。这种方式被称为**“空闲列表”（Free List）**。[^2]

> [!NOTE] 取长补短
> 虽然说采用类似于 CMS 收集器时采用的是较复杂的空闲列表法，但为了更快地分配，CMS 设计了一个叫做 `Linear Allocation Buffer` 的分配缓冲区：缓冲区之间由空闲列表管理，而缓冲区内部则使用指针碰撞方式分配内存。
> 而对于更现代的 JVM 来说，则会使用更为复杂的 **Region-based** 内存管理架构，思想上也是二种方式的取长补短。

但仅仅如此也不足以应对虚拟机中无时无刻不在进行的对象创建行为——以指针碰撞方法为例，如果 A 线程读取到了指针的位置，在将指针的位置移动之前，B 线程此时也加入对象创建，那么 B 线程很可能读取到的是已经过时的指针位置，于是产生了冲突。

为了解决分配内存时出现的并发冲突，通常采用**两层**防御体系：
1. **本地线程分配缓冲（Thread Local Allocation Buffer，TLAB）**，即每个线程在 Java 堆中预先分配一小块内存。当线程需要分配内存时，先在线程的本地缓冲区中分配，只有本地缓冲区用完了，才会去共享区分配内存，此时需要同步锁定。
 TLAB 的关键参数可见下表：

| 参数                           | 默认值   | 说明                |
| ---------------------------- | ----- | ----------------- |
| `-XX:+UseTLAB`               | true  | 启用 TLAB            |
| `-XX:TLABSize`               | 动态计算  | 初始大小（通常几十 KB~几百 KB） |
| `-XX:MinTLABSize`            | 2 KB   | 最小 TLAB            |
| `-XX:MaxTLABSize`            | 256 KB | 最大 TLAB（受限于 Eden）   |
| `-XX:TLABWasteTargetPercent` | 1%    | TLAB 浪费率目标         |
| `-XX:+ResizeTLAB`            | true  | 允许动态调整大小          |
2. **CAS** 原子操作 + 自旋重试，保证更新操作的原子性。

#### 1.2 初始化零值
内存分配完成之后，虚拟机必须将分配到的内存空间（这一步不包括对象头）都初始化为**零值**。[^3]这步操作保证了对象的实例字段在 Java 代码中可以不赋初始值就直接使用，使程序能访问到这些字段的数据类型所对应的零值。

#### 1.3 设置对象头
接下来，Java 虚拟机还要对对象的**对象头（Object Header）**进行必要的设置，比如 `Mark Word` 和 `Klass Pointer`， 这些信息可以算是对象在虚拟机中的“身份证”。

#### 1.4 构造函数
以上工作完成后，从虚拟机的视角来看，对象已经构建完成。但是别忘了在 Java 程序中，对象创建才刚开始——构造函数还未被执行，所有的字段还只是零值。一般来说在 `new` 指令之后会接着执行 `<init>()` 方法，这样一个真正可用的对象才算完全被构造出来。


### 2. 对象的内存布局
HotSpot 虚拟机里，对象在堆内存中的存储布局可以划分为三个部分：
- ** [[#2.1 对象头|对象头（Header）]] **
- **实例数据（Instance Data）**
- **对齐填充（Padding）**

![image.png|457](https://pic-1371809842.cos.ap-chengdu.myqcloud.com/PicGo/20260313004121430.png?imageSlim)

#### 2.1 对象头
对象头部分主要包括两类信息。

- 第一类是 `Mark Word` 位图，用于存储对象自身的运行时数据：如哈希码（HashCode）、GC 分代年龄、锁状态标志、线程持有的锁、偏向线程 ID、偏向时间戳等。未开启压缩指针[^4]的情况下，这部分数据的长度在 32 位和 64 位的虚拟机中分别为 32 个比特和 64 个比特。
- 第二类为类型指针 `Klass Pointer` ，即对象指向它的类型元数据的指针，Java 虚拟机通过这个指针来确定该对象是哪个类的实例。未开启指针压缩时占用 8 个字节，开启后占用 4 字节。

另外的，如果是数组对象，则有额外的 4 个字节存放数组长度。

#### 2.2 实例数据
实例数据部分是对象真正存储的有效信息，包含我们在程序代码里面所定义的各种类型的字段内容。这一段数据的长度基本上是多少就是多少，如果想压缩的话，可以让子类中较短的变量插入父类变量的空隙中（`+XX：CompactFields = true`，不过这个参数默认就是开着的）以节省一点点空间。

#### 2.3 对齐填充
正如其名，这块内存空间只是为了对齐与填充，并无实际作用。但实际上，HotSpot 虚拟机的自动内存管理系统要求对象起始地址必须是 8 字节的整数倍，也就是说 Java 对象的大小必须是 8 字节的倍数。如果实例数据部分没有对齐上 8 字节，那么就需要额外浪费一些空间来填充。

> [!NOTE] 取舍之道
> Java 的 8 字节对齐填充，就像是内存的 8 bit 为一组变为 1 byte 一样。这可以让 32 位长度指针表示出 32 G （4 G \* 8）的内存空间。对象头中的类型指针并不指向真实的操作系统内存地址，而是以 8 字节为单位映射之后的虚拟机地址。所以相对于操作系统的等长指针可以进行 8 倍的扩容。 
> 对于大部分处理器，这种对齐方案都是最优的，在 8 字节对齐的位置，对象可以更快地访问。
>   
>  *Q：那为什么不采用能表示出 64 G 内存空间的 16 字节对齐呢？*
> A：因为在那种情况下，在堆中压缩指针所节约的成本就 cover 不住由于对齐对象而浪费的内存了。

实际应用中不建议将 JVM 的堆内存设置为大于 32 G。32 G 是一个大致的临界值，超过之后 JVM 就无法使用压缩指针了（还想用压缩的话就得手动设置 `-XX:ObjectAlignmentInBytes=16`），对象体积膨胀（在小对象占多时尤其明显），CPU 可缓存的对象减少，传统 GC 收集器压力也会增大（在 JDK 11+ 可以通过参数 `-XX:+UseZGC` 使用 ZGC 开启大堆内存 GC 优化），最终导致程序性能下降。


### 3. 对象的访问与定位
Java 程序通过 [[JVM 运行时内存区#^516a4d | 栈]]上的 reference 数据来操作堆上的具体对象。而具体的对象访问方式则由虚拟机决定，主流可以分为两种方式：**句柄**和**直接指针**。
- 如果使用句柄访问的话，Java 堆中将会划分出一块内存来作为句柄池，reference 中存储的就是对象的句柄地址，而句柄中包含了对象实例数据与类型数据各自具体的地址信息。如下图所示。

![句柄访问|705](https://pic-1371809842.cos.ap-chengdu.myqcloud.com/PicGo/20260326000623619.png?imageSlim)

- 如果使用直接指针访问的话，reference 中存储的直接就是对象地址，如果只是访问对象本身的话，就不需要多一次间接访问的开销。如下图所示。

![直接指针](https://pic-1371809842.cos.ap-chengdu.myqcloud.com/PicGo/20260326001117413.png?imageSlim)

两种方式[^5]各有优劣。使用句柄访问的好处就是 reference 中只需要存储句柄地址，在对象被移动（比如垃圾收集时移动对象）时只会改变句柄中的实例数据指针，而 reference 本身不需要被修改；而直接指针则因为节省了一次指针定位开销，有更快的访问速度，长期积少成多后也是一项很客观的时间成本。




[^1]: 至于怎么找到这块“合适”的空间，可以采用很多方法如首次适应、最佳适应和最坏适应等，这并不是我们讨论的重点。

[^2]: 当然，操作系统中不止这一种方法，还有空闲链表、位图、成组链接等方法 。

[^3]: 如果使用了 TLAB 的话，这一项工作也可以提前至 TLAB 分配时顺便进行。

[^4]: 实际上压缩指针影响的是 `Klass Pointer`，而不是 `Mark Word`。当堆内存小于 4 G 时，不需要启用指针压缩，jvm 会直接去除高 32 位地址（因为根本用不到高位地址，算一算就知道了），只使用低虚拟地址空间；而当堆内存大于 32 G 时，压缩指针会失效，会强制使用 64 位(即 8 字节)来对 java 对象寻址。

[^5]: 以 HotSpot 虚拟机为例，它主要使用第二种方式（即直接指针）进行对象访问。
