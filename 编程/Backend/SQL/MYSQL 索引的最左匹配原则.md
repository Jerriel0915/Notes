MySQL 索引的**最左前缀匹配原则（Leftmost Prefix Rule）**是联合索引（Composite Index）使用的核心规则，它决定了查询如何有效利用多列索引。

## 1. 核心原理
对于联合索引 `(a, b, c)`，MySQL 会从索引的最左边开始，按顺序匹配查询条件。只有当查询条件**从索引的第一列开始连续匹配**时，才能使用该索引。

### 匹配规则

| 索引列顺序 | 有效查询示例 | 是否使用索引 | 说明 |
|-----------|------------|------------|------|
| `(a, b, c)` | `WHERE a = 1` | ✅ 是 | 使用索引第 1 列 |
| `(a, b, c)` | `WHERE a = 1 AND b = 2` | ✅ 是 | 使用索引前 2 列 |
| `(a, b, c)` | `WHERE a = 1 AND b = 2 AND c = 3` | ✅ 是 | 完全使用索引 |
| `(a, b, c)` | `WHERE b = 2` | ❌ 否 | 缺少第 1 列 `a` |
| `(a, b, c)` | `WHERE a = 1 AND c = 3` | ⚠️ 部分 | 仅使用 `a`，`c` 无法使用（中间 `b` 断了） |
| `(a, b, c)` | `WHERE b = 2 AND c = 3` | ❌ 否 | 缺少第 1 列 `a` |

## 2. 关键要点
### 2.1 **顺序至关重要**
索引列的定义顺序 `(a, b, c)` **不等于**查询条件的书写顺序，但查询条件**必须包含最左列**才能触发索引。

```sql
-- 索引 (a, b, c)
WHERE b = 2 AND a = 1  -- ✅ 有效！优化器会自动调整为 a=1 优先
WHERE c = 3 AND b = 2 AND a = 1  -- ✅ 有效！
```

### 2.2  **范围查询会阻断后续列**
当某列使用范围查询（`>`、`<`、`BETWEEN`、`LIKE '%x'`）时，该列右侧的索引列将失效：

```sql
-- 索引 (a, b, c)
WHERE a = 1 AND b > 2 AND c = 3  
-- 结果：只用 (a, b)，c 不走索引（b 是范围查询，c 被阻断）
```

### 2.3 **等值查询 + IN 不会阻断**
`IN` 被视为多个等值查询，不会阻断后续列：

```sql
WHERE a = 1 AND b IN (2, 3) AND c = 4  
-- 结果：a, b, c 都走索引
```

## 3. 为何如此
### 3.1 **B+树的排序本质**
InnoDB 使用 B+树作为索引结构，其核心特性是**数据按索引列顺序物理排序存储**：

```plain
联合索引 (a, b, c) 的 B+树结构：

                    [10, 100, 5]
                   /      |      \
        [1, 50, 3]    [10, 80, 2]    [10, 100, 5]    [20, 200, 8]
          ...            ...             ...             ...
```

- 首先按 `a` 排序
- `a` 相同则按 `b` 排序
- `a`、`b` 都相同再按 `c` 排序

### 3.2 **为什么必须"最左"开始？**
B+树的查找过程是**从根节点到叶子节点的二分查找**，它要求比较条件必须是**确定的前缀**：

| 查询条件                | 能否定位 | 原因                          |
| :------------------ | :--- | :-------------------------- |
| `a = 10`            | ✅ 能  | 知道从哪个分支开始找                  |
| `a = 10 AND b = 80` | ✅ 能  | 在 `a=10` 的子树下找 `b=80`       |
| `b = 80`            | ❌ 不能 | `b` 在第二层，分散在整个树中，不知道第一层走哪条路 |

**类比**：就像查字典，你知道拼音第二个字母是 `h`，但不知道第一个字母，无法直接定位。

## 4. 实际示例
假设有表 `employees` 和索引 `idx_dept_salary_age (department, salary, age)`：

```sql
-- ✅ 完全匹配，性能最佳
SELECT * FROM employees 
WHERE department = 'Tech' AND salary = 50000 AND age = 30;

-- ✅ 匹配前两列
SELECT * FROM employees 
WHERE department = 'Tech' AND salary > 40000;

-- ⚠️ 只匹配 department，salary 范围查询导致 age 失效
SELECT * FROM employees 
WHERE department = 'Tech' AND salary > 40000 AND age > 25;

-- ❌ 完全不走索引（缺少 department）
SELECT * FROM employees 
WHERE salary = 50000 AND age = 30;

-- ✅ 走索引（优化器调整顺序）
SELECT * FROM employees 
WHERE salary = 50000 AND department = 'Tech';
```

## 5. 设计建议

1. **高频查询列放左边**：将区分度高、查询频繁的列放在索引左侧
2. **避免范围查询在靠前位置**：如果 `b` 常做范围查询，考虑索引顺序 `(a, c, b)` 或建立单独索引
3. **覆盖索引优化**：查询的列都在索引中时，无需回表查询
